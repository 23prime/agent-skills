# Creating Skills

効果的なスキルを作成するためのガイド。詳細なベストプラクティスは
[references/best-practices.ja.md](references/best-practices.ja.md) を参照。

## クイックリファレンス

### スキル構造

```txt
skill-name/
├── SKILL.md              # 必須: YAML frontmatter + 指示
├── scripts/              # 任意: 実行可能コード (Python/Bash)
├── references/           # 任意: 必要に応じて読み込むドキュメント
└── assets/               # 任意: 出力用のテンプレート、画像、フォント
```

### YAML Frontmatter

**name**: 最大 64 文字、小文字/数字/ハイフンのみ、動名詞形を推奨
（`processing-pdfs` であって `helper` ではない）

**description**: 最大 1024 文字、三人称、WHAT と WHEN の両方を含める

### 主要原則

- **簡潔さが鍵**: Claude が既に知らないコンテキストのみを追加する
- **重複を避ける**: 情報は SKILL.md か references のどちらか一方に配置する
- **SKILL.md は 500 行未満**: 詳細は references/ に移動する
- **参照は 1 階層**: ネストしたファイル参照を避ける

## スキル作成プロセス

### ステップ 1: 具体例によるスキルの理解

スキルの使用パターンが既に明確に理解されている場合のみスキップする。既存スキルを扱う場合でも有用なステップである。

効果的なスキルを作成するには、スキルがどのように使われるかの具体例を明確に理解する。この理解は、ユーザーからの直接的な例か、ユーザーのフィードバックで検証された生成例から得られる。

例えば、image-editor スキルを構築する場合、関連する質問は以下の通り:

- 「image-editor スキルはどのような機能をサポートすべきか？ 編集、回転、他には？」
- 「このスキルがどのように使われるか、いくつか例を挙げてもらえるか？」
- 「『この画像の赤目を除去して』や『この画像を回転して』のようなリクエストが想像できるが、他にどのような使い方を想定しているか？」
- 「どのような発言がこのスキルをトリガーすべきか？」

ユーザーを圧倒しないよう、1 つのメッセージで質問を多くしすぎない。最も重要な質問から始め、必要に応じてフォローアップする。

スキルがサポートすべき機能が明確に把握できた時点でこのステップを完了する。

### ステップ 2: 再利用可能なスキルコンテンツの計画

具体例を効果的なスキルに変えるために、各例を以下の観点で分析する:

1. その例をゼロから実行する方法を検討する
2. これらのワークフローを繰り返し実行する際に役立つスクリプト、リファレンス、アセットを特定する

例: 「この PDF を回転して」のようなクエリを処理する `pdf-editor` スキルを構築する場合の分析:

1. PDF の回転は毎回同じコードを書き直す必要がある
2. `scripts/rotate_pdf.py` スクリプトをスキルに格納すると有用

例: 「TODO アプリを作って」や「歩数を追跡するダッシュボードを作って」のようなクエリに対応する `frontend-webapp-builder` スキルを設計する場合の分析:

1. フロントエンド Web アプリの作成は毎回同じボイラープレート HTML/React を必要とする
2. ボイラープレート HTML/React プロジェクトファイルを含む `assets/hello-world/` テンプレートをスキルに格納すると有用

例: 「今日ログインしたユーザーは何人か？」のようなクエリを処理する `big-query` スキルを構築する場合の分析:

1. BigQuery のクエリは毎回テーブルスキーマとリレーションシップを再調査する必要がある
2. テーブルスキーマを文書化した `references/schema.md` ファイルをスキルに格納すると有用

スキルのコンテンツを確立するために、各具体例を分析し、含めるべき再利用可能リソース（スクリプト、リファレンス、アセット）のリストを作成する。

### ステップ 3: スキルの初期化

この時点で実際にスキルを作成する。

開発中のスキルが既に存在し、イテレーションやパッケージングが必要な場合のみスキップする。その場合は次のステップに進む。

新しいスキルをゼロから作成する場合は、必ず `init_skill.py` スクリプトを実行する。このスクリプトはスキルに必要なすべてを自動的に含むテンプレートスキルディレクトリを生成し、スキル作成プロセスをより効率的かつ信頼性の高いものにする。

使い方:

```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

このスクリプトは:

- 指定パスにスキルディレクトリを作成する
- 適切な frontmatter と TODO プレースホルダーを含む SKILL.md テンプレートを生成する
- サンプルリソースディレクトリ `scripts/`、`references/`、`assets/` を作成する
- 各ディレクトリにカスタマイズまたは削除可能なサンプルファイルを追加する

初期化後、生成された SKILL.md とサンプルファイルを必要に応じてカスタマイズまたは削除する。

### ステップ 4: スキルの編集

（新規生成または既存の）スキルを編集する際は、スキルが別の Claude インスタンスによって使用されることを意識する。Claude にとって有益かつ自明でない情報を含めることに注力する。別の Claude インスタンスがこれらのタスクをより効果的に実行するために、どのような手続き的知識、ドメイン固有の詳細、再利用可能なアセットが役立つかを検討する。

#### 再利用可能なスキルコンテンツから始める

実装を開始するには、上記で特定した再利用可能リソース（`scripts/`、`references/`、`assets/` ファイル）から着手する。このステップではユーザーの入力が必要になる場合がある。例えば、`brand-guidelines` スキルを実装する際、ユーザーが `assets/` に格納するブランドアセットやテンプレート、または `references/` に格納するドキュメントを提供する必要がある場合がある。

また、スキルに不要なサンプルファイルやディレクトリを削除する。初期化スクリプトは構造を示すために `scripts/`、`references/`、`assets/` にサンプルファイルを作成するが、ほとんどのスキルではすべてが必要になるわけではない。

#### SKILL.md の更新

SKILL.md を完成させるために、以下の質問に回答する:

1. スキルの目的は何か（数文で）
2. どのような場合にスキルを使用すべきか
3. 実際に Claude はスキルをどのように使用すべきか。すべてのバンドルリソースを参照する

コンテンツガイドライン、一般的なパターン、アンチパターンについては
[references/best-practices.ja.md](references/best-practices.ja.md) を参照。

### ステップ 5: スキルのバリデーション

スキルの準備が整ったら、すべての要件を満たしていることをバリデーションで確認する:

```bash
scripts/quick_validate.py <path/to/skill-folder>
```

チェック項目:

- YAML frontmatter の形式と必須フィールド
- スキルの命名規則とディレクトリ構造
- description の完全性と品質
- ファイル構成とリソース参照

バリデーションが失敗した場合、報告されたエラーを修正し再度コマンドを実行する。

### ステップ 6: スキルの lint

Markdown リンターを実行し、スキル内の SKILL.md およびその他の Markdown ファイルを自動修正・検証する:

```bash
mise run fix-md
mise run check-md
```

自動修正後もエラーが残る場合は、`check-md` がエラーゼロになるまで手動で修正する。

### ステップ 7: 評価とイテレーション

**まず評価を構築する**: 広範なドキュメントを書く前に評価を作成する。これにより、スキルが想像上の問題ではなく実際の問題を解決することを保証する。

**評価駆動開発:**

1. **ギャップを特定**: スキルなしで代表的なタスクを Claude に実行させ、失敗を記録する
2. **評価を作成**: これらのギャップをテストする 3 つ以上のシナリオを構築する
3. **ベースラインを確立**: スキルなしの Claude のパフォーマンスを測定する
4. **最小限の指示を書く**: 評価をパスするのに十分なコンテンツのみを作成する
5. **イテレーション**: 評価を実行し、ベースラインと比較し、改良する

**Claude によるイテレーション開発:**

ある Claude インスタンス（「Claude A」）でスキルを作成し、別のインスタンス（「Claude B」）でテストする:

1. スキルなしでタスクを完了し、繰り返し提供されたコンテキストをメモする
2. Claude A に再利用可能なパターンを捕捉するスキルの作成を依頼する
3. 簡潔さを確認 - Claude が既に知っている説明を削除する
4. Claude B で類似タスクをテストし、挙動を観察する
5. 具体的な観察結果を持って Claude A に戻り改善を依頼する
6. 観察-改良-テストのサイクルを繰り返す

**Claude のスキルナビゲーションを観察する:**

- 予期しない探索パスは直感的でない構造を示す可能性がある
- 見落とされたリファレンスはリンクをより明示的にする必要を示す
- 特定ファイルへの過度な依存はコンテンツを SKILL.md に移動すべきことを示す
- 無視されるファイルは不要か、シグナルが不十分な可能性がある
