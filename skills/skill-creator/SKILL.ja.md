# Skill Creator

効果的なスキルを作成するためのガイドを提供するスキル。

## スキルについて

スキルは、専門知識・ワークフロー・ツールを提供することで Claude の能力を拡張する、モジュール化された自己完結型パッケージである。特定のドメインやタスクに対する「オンボーディングガイド」のようなもので、汎用エージェントとしての Claude を、どのモデルも完全には持ち得ない手続き的知識を備えた専門エージェントへと変える。

### スキルが提供するもの

1. 専門ワークフロー - 特定ドメイン向けの複数ステップの手順
2. ツール連携 - 特定のファイル形式や API を扱うための手順
3. ドメイン知識 - 企業固有の知識、スキーマ、ビジネスロジック
4. バンドルリソース - 複雑で反復的なタスク向けのスクリプト、リファレンス、アセット

### スキルの構成

すべてのスキルは必須の SKILL.md とオプションのバンドルリソースで構成される:

```txt
skill-name/
├── SKILL.md (必須)
│   ├── YAML frontmatter メタデータ (必須)
│   │   ├── name: (必須)
│   │   └── description: (必須)
│   └── Markdown による指示 (必須)
└── バンドルリソース (任意)
    ├── scripts/          - 実行可能コード (Python/Bash 等)
    ├── references/       - 必要に応じてコンテキストに読み込むドキュメント
    └── assets/           - 出力に使用するファイル (テンプレート、アイコン、フォント等)
```

#### SKILL.md (必須)

**メタデータの品質:** YAML frontmatter の `name` と `description` は、Claude がそのスキルをいつ使うかを決定する。スキルの内容と使用タイミングを具体的に記述する。三人称を使用する（例: 「Use this skill when...」ではなく「This skill should be used when...」）。

#### バンドルリソース (任意)

##### スクリプト (`scripts/`)

決定論的な信頼性が求められるタスクや、繰り返し書き直されるタスク向けの実行可能コード (Python/Bash 等)。

- **含めるべき場合**: 同じコードが繰り返し書き直される場合や、決定論的な信頼性が必要な場合
- **例**: PDF 回転タスク用の `scripts/rotate_pdf.py`
- **利点**: トークン効率が良く、決定論的で、コンテキストに読み込まずに実行可能
- **注意**: スクリプトのパッチ適用や環境固有の調整のために Claude が読み込む必要がある場合もある

##### リファレンス (`references/`)

Claude の処理や思考に情報を提供するために、必要に応じてコンテキストに読み込むドキュメントおよび参考資料。

- **含めるべき場合**: Claude が作業中に参照すべきドキュメントがある場合
- **例**: 財務スキーマ用の `references/finance.md`、NDA テンプレート用の `references/mnda.md`、社内ポリシー用の `references/policies.md`、API 仕様用の `references/api_docs.md`
- **用途**: データベーススキーマ、API ドキュメント、ドメイン知識、社内ポリシー、詳細なワークフローガイド
- **利点**: SKILL.md を簡潔に保ち、Claude が必要と判断した場合のみ読み込まれる
- **ベストプラクティス**: ファイルが大きい場合 (1 万語超) は、SKILL.md に grep 検索パターンを含める
- **重複の回避**: 情報は SKILL.md かリファレンスファイルのどちらか一方にのみ記載する。スキルの本質に関わる情報でない限り、詳細情報はリファレンスファイルに配置する。これにより SKILL.md を簡潔に保ちつつ、コンテキストウィンドウを圧迫せずに情報を発見可能にする。SKILL.md には本質的な手順とワークフローガイダンスのみを残し、詳細な参考資料、スキーマ、例はリファレンスファイルに移動する。

##### アセット (`assets/`)

コンテキストに読み込むことを意図せず、Claude が生成する出力内で使用するファイル。

- **含めるべき場合**: スキルが最終出力で使用するファイルがある場合
- **例**: ブランドアセット用の `assets/logo.png`、PowerPoint テンプレート用の `assets/slides.pptx`、HTML/React ボイラープレート用の `assets/frontend-template/`、タイポグラフィ用の `assets/font.ttf`
- **用途**: テンプレート、画像、アイコン、ボイラープレートコード、フォント、コピーや修正の対象となるサンプルドキュメント
- **利点**: 出力リソースをドキュメントから分離し、Claude がコンテキストに読み込まずにファイルを使用可能にする

### プログレッシブディスクロージャの設計原則

スキルはコンテキストを効率的に管理するために 3 段階の読み込みシステムを使用する:

1. **メタデータ (name + description)** - 常にコンテキスト内 (約 100 語)
2. **SKILL.md 本文** - スキルがトリガーされた時 (5,000 語未満)
3. **バンドルリソース** - Claude が必要とした時 (無制限*)

*スクリプトはコンテキストウィンドウに読み込まずに実行できるため無制限。

## スキル作成プロセス

スキルを作成するには、「スキル作成プロセス」を順番に実行する。明確に該当しない理由がある場合のみステップをスキップする。

### ステップ 1: 具体例によるスキルの理解

スキルの使用パターンが既に明確に理解されている場合のみスキップする。既存スキルを扱う場合でも有用なステップである。

効果的なスキルを作成するには、スキルがどのように使われるかの具体例を明確に理解する。この理解は、ユーザーからの直接的な例か、ユーザーのフィードバックで検証された生成例から得られる。

例えば、image-editor スキルを構築する場合、関連する質問は以下の通り:

- 「image-editor スキルはどのような機能をサポートすべきか？ 編集、回転、他には？」
- 「このスキルがどのように使われるか、いくつか例を挙げてもらえるか？」
- 「『この画像の赤目を除去して』や『この画像を回転して』のようなリクエストが想像できるが、他にどのような使い方を想定しているか？」
- 「どのような発言がこのスキルをトリガーすべきか？」

ユーザーを圧倒しないよう、1 つのメッセージで質問を多くしすぎない。最も重要な質問から始め、必要に応じてフォローアップする。

スキルがサポートすべき機能が明確に把握できた時点でこのステップを完了する。

### ステップ 2: 再利用可能なスキルコンテンツの計画

具体例を効果的なスキルに変えるために、各例を以下の観点で分析する:

1. その例をゼロから実行する方法を検討する
2. これらのワークフローを繰り返し実行する際に役立つスクリプト、リファレンス、アセットを特定する

例: 「この PDF を回転して」のようなクエリを処理する `pdf-editor` スキルを構築する場合の分析:

1. PDF の回転は毎回同じコードを書き直す必要がある
2. `scripts/rotate_pdf.py` スクリプトをスキルに格納すると有用

例: 「TODO アプリを作って」や「歩数を追跡するダッシュボードを作って」のようなクエリに対応する `frontend-webapp-builder` スキルを設計する場合の分析:

1. フロントエンド Web アプリの作成は毎回同じボイラープレート HTML/React を必要とする
2. ボイラープレート HTML/React プロジェクトファイルを含む `assets/hello-world/` テンプレートをスキルに格納すると有用

例: 「今日ログインしたユーザーは何人か？」のようなクエリを処理する `big-query` スキルを構築する場合の分析:

1. BigQuery のクエリは毎回テーブルスキーマとリレーションシップを再調査する必要がある
2. テーブルスキーマを文書化した `references/schema.md` ファイルをスキルに格納すると有用

スキルのコンテンツを確立するために、各具体例を分析し、含めるべき再利用可能リソース（スクリプト、リファレンス、アセット）のリストを作成する。

### ステップ 3: スキルの初期化

この時点で実際にスキルを作成する。

開発中のスキルが既に存在し、イテレーションやパッケージングが必要な場合のみスキップする。その場合は次のステップに進む。

新しいスキルをゼロから作成する場合は、必ず `init_skill.py` スクリプトを実行する。このスクリプトはスキルに必要なすべてを自動的に含むテンプレートスキルディレクトリを生成し、スキル作成プロセスをより効率的かつ信頼性の高いものにする。

使い方:

```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

このスクリプトは:

- 指定パスにスキルディレクトリを作成する
- 適切な frontmatter と TODO プレースホルダーを含む SKILL.md テンプレートを生成する
- サンプルリソースディレクトリ `scripts/`、`references/`、`assets/` を作成する
- 各ディレクトリにカスタマイズまたは削除可能なサンプルファイルを追加する

初期化後、生成された SKILL.md とサンプルファイルを必要に応じてカスタマイズまたは削除する。

### ステップ 4: スキルの編集

（新規生成または既存の）スキルを編集する際は、スキルが別の Claude インスタンスによって使用されることを意識する。Claude にとって有益かつ自明でない情報を含めることに注力する。別の Claude インスタンスがこれらのタスクをより効果的に実行するために、どのような手続き的知識、ドメイン固有の詳細、再利用可能なアセットが役立つかを検討する。

#### 再利用可能なスキルコンテンツから始める

実装を開始するには、上記で特定した再利用可能リソース（`scripts/`、`references/`、`assets/` ファイル）から着手する。このステップではユーザーの入力が必要になる場合がある。例えば、`brand-guidelines` スキルを実装する際、ユーザーが `assets/` に格納するブランドアセットやテンプレート、または `references/` に格納するドキュメントを提供する必要がある場合がある。

また、スキルに不要なサンプルファイルやディレクトリを削除する。初期化スクリプトは構造を示すために `scripts/`、`references/`、`assets/` にサンプルファイルを作成するが、ほとんどのスキルではすべてが必要になるわけではない。

#### SKILL.md の更新

**文体:** スキル全体を**命令形/不定詞形**（動詞先頭の指示）で記述し、二人称は使わない。客観的で指示的な言い方を使用する（例: 「You should do X」や「If you need to do X」ではなく「To accomplish X, do Y」）。これにより AI が消費する際の一貫性と明瞭性を維持する。

SKILL.md を完成させるために、以下の質問に回答する:

1. スキルの目的は何か（数文で）
2. どのような場合にスキルを使用すべきか
3. 実際に Claude はスキルをどのように使用すべきか。上記で開発したすべての再利用可能スキルコンテンツを参照し、Claude がそれらの使い方を把握できるようにする

### ステップ 5: スキルのバリデーション

スキルの準備が整ったら、すべての要件を満たしていることをバリデーションで確認する:

```bash
scripts/quick_validate.py <path/to/skill-folder>
```

チェック項目:

- YAML frontmatter の形式と必須フィールド
- スキルの命名規則とディレクトリ構造
- description の完全性と品質
- ファイル構成とリソース参照

バリデーションが失敗した場合、報告されたエラーを修正し再度コマンドを実行する。

### ステップ 6: イテレーション

スキルのテスト後、ユーザーが改善を依頼する場合がある。これはスキルの使用直後、スキルのパフォーマンスに関する新鮮なコンテキストがある状態で行われることが多い。

**イテレーションのワークフロー:**

1. 実際のタスクでスキルを使用する
2. 困難や非効率に気づく
3. SKILL.md やバンドルリソースをどのように更新すべきかを特定する
4. 変更を実装し再度テストする
